# Evaluation Plan for IoT InfraLab (MVP)

This document outlines the test cases, procedures, and pass/fail criteria for evaluating the successful implementation of the Minimum Viable Product (MVP).

## Objective

To verify that the system can successfully simulate an IoT environment, execute a defined attack, detect that attack, and visualize both the normal operation and the security alert in a single, end-to-end workflow.

---

## Test Suite: End-to-End MVP Scenario

This suite consists of a single, continuous test divided into four sequential test cases. Each test case must pass for the evaluation to be considered successful.

### Test Case 1: System Deployment and Stability

*   **Description:** Verifies that all required services for the MVP can be started correctly and remain stable.
*   **Steps:**
    1.  Open a terminal in the project root directory.
    2.  Run the command: `docker compose up -d`
    3.  Wait for 2 minutes for all services to initialize.
    4.  Run the command: `docker compose ps`
*   **Expected Result:** The `docker compose ps` command shows all MVP-required services (`mosquitto`, `nodered`, `influxdb`, `grafana`, `loki`, `promtail`, `suricata`) with a status of `Up` or `running`. No services should be in a `restarting` or `exited` state.
*   **Pass/Fail Criteria:**
    *   **PASS:** All required services are running without errors.
    *   **FAIL:** Any required service fails to start or enters a crash loop.

### Test Case 2: Sensor Simulation

*   **Description:** Verifies that a simulated sensor can be created via Node-RED.
*   **Steps:**
    1.  Access the Node-RED UI at `http://localhost:1880`.
    2.  Use the "Node Management" flow to trigger the creation of one simulated sensor container.
    3.  Run the command `docker ps` in a terminal.
*   **Expected Result:** The `docker ps` output shows a new container running with an image name related to `docker_sensor`.
*   **Pass/Fail Criteria:**
    *   **PASS:** A new sensor container is created and running within 20 seconds.
    *   **FAIL:** The container is not created or fails to start.

### Test Case 3: Attack Execution and IDS Detection

*   **Description:** Verifies that the defined attack can be executed and that the Suricata IDS correctly identifies it and logs an alert.
*   **Steps:**
    1.  From the Node-RED "Cyber Attack" flow, trigger the "MQTT DoS" attack.
    2.  Wait for 15 seconds.
    3.  Inspect the Suricata log file by running:
        `cat infrastructure/suricata/logs/eve.json | jq 'select(.event_type=="alert")'`
*   **Expected Result:** The `jq` command output shows at least one JSON object representing a security alert. The `alert.signature` field in the JSON must be exactly `"DoS Attempt: MQTT SYN Flood Detected"`.
*   **Pass/Fail Criteria:**
    *   **PASS:** An alert with the correct signature is present in `eve.json`.
    *   **FAIL:** No alert is generated, or the alert generated has an incorrect signature.

### Test Case 4: Dashboard Interactivity Guide

*   **Description:** Verifies the presence of a user-guided tour for key dashboards, explaining UI elements and their functions.
*   **Steps:**
    1.  Access the Grafana UI at `http://localhost:3000`.
    2.  Navigate to the "Node Management", "Cyber Attack", and "Security Operations (SOC)" dashboards.
    3.  Check for a "guide" or "tour" feature (e.g., a button that initiates a step-by-step popup walkthrough).
*   **Expected Result:** Each dashboard provides an interactive tour that highlights and explains the purpose of its panels and controls.
*   **Pass/Fail Criteria:**
    *   **PASS:** An interactive guide is present and functional on all three specified dashboards.
    *   **FAIL:** No such guide is available.

### Test Case 5: IoT Data Visualization

*   **Description:** Verifies that live sensor data is correctly ingested and displayed in the "IoT Sensor Dashboard".
*   **Steps:**
    1.  Ensure at least one sensor is running (from Test Case 2).
    2.  Access the Grafana UI at `http://localhost:3000`.
    3.  Navigate to the "IoT Sensor Dashboard".
    4.  Observe the panels (e.g., "Total Sensors", "Sensor Telemetry", "Network Traffic").
*   **Expected Result:** The dashboard panels show live, updating data points corresponding to the telemetry from active sensors. Charts should update, and counters should reflect the correct number of sensors.
*   **Pass/Fail Criteria:**
    *   **PASS:** All panels display accurate, updating telemetry from the sensors within 30 seconds of the dashboard loading.
    *   **FAIL:** Panels are empty, show errors, or do not update with new data.

### Test Case 6: Alert Visualization

*   **Description:** Verifies that security alerts generated by Suricata are collected and displayed in the "Security Operations (SOC)" dashboard.
*   **Steps:**
    1.  After a successful attack in Test Case 3, navigate to the "Security Operations (SOC)" dashboard in Grafana.
    2.  Observe the "Suricata Alerts" and "Alerts by Signature" panels.
*   **Expected Result:** The alert panels update to show the new alert detected in Test Case 3. The alert signature `"DoS Attempt: MQTT SYN Flood Detected"` should be visible.
*   **Pass/Fail Criteria:**
    *   **PASS:** The alert is visible in the Grafana dashboard within 30 seconds of the attack.
    *   **FAIL:** The alert does not appear in the dashboard.
